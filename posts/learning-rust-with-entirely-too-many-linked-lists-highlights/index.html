This is a “learning cheatsheet” of Learn Rust With Entirely Too Many Linked Lists, which you can use as a way to test yourself and do some spaced repetition to memorize the concepts and highlights from it.

Why?

Whenever I start programming in Rust, I get “stuck” somewhere around the Box keyword and just give up. This year, I’m planning to do some Rust programming (even if just for fun) and I REALLY want to nail down the fundamentals. Gotta walk before I can run.

How?

By using my own flashcards!

pub for “public” says we want people outside of a module to be able to use the thing. In Go, we would have just Capitalized the thing, which is perfect.

Box documentation.

Box<T>, casually referred to as a ‘box’, provides the simplest form of heap allocation in Rust. Boxes provide ownership for this allocation, and drop their contents when they go out of scope.

OK, so reminds me of an smart pointer in C++.

A sum type: a type that can have different values which may be different types.

Because of this property, enums are laid out in memory in such a way that they need enough space to store the largest of their variants (+ alignment padding).

Enums let us declare a type that can contain one of seceral values, structs let us declare a type that contains many values at once.
impl is how we associate actual code with a type. So a way to create methods for a specific type, or implement a trait for a type.

Functions:

fn function_name(param1: Type1, param2: Type2) -> ReturnType {
  // function body
}

Methods:

fn foo(self, arg2: Type2) -> ReturnType {
  // body
}
No need for a return keyword in Rust, the last expression in a function is implicitly returned.
  • self - Value: “True” ownership (you can destroy it)
  • &mut self - Mutable reference: Temporary exclusive access (you can mutate it)
  • &self - Shared reference: Temporary shared access (you can read it, but not modify it)

Interesting note here, the book says:

> cargo build
error[E0507]: cannot move out of borrowed content
  --> src/first.rs:19:19
   |
19 |             next: self.head,
   |                   ^^^^^^^^^ cannot move out of borrowed content

While I see:

error[E0507]: cannot move out of `self.head` which is behind a mutable reference
  --> src/first.rs:23:19
   |
23 |             next: self.head,
   |                   ^^^^^^^^^ move occurs because `self.head` has type `Link`, which does not implement the `Copy` trait

Seems like the error messages from the compiler have gotten a lot friendlier, but in this case, they would have led my down the wrong path; I don’t want to implement the Copy trait to implement push here, I want to actually change self.head to first point to Link::Empty (as I build new_node to point at what used to be there) and after new_node has been built I want to change it to new_node.

If we got a mutable reference, that means we don’t have “true” ownership; we have temporary exclusive access, which means we can mutate. But we specifically cannot destroy whatever we have a mutable reference to.

We would like to:

fn (&mut thingy) {
  // This causes a move and is not allowed
  new_ownership = thingy
  thingy = i_put_it_right_back_i_swear
}

But that won’t fly, since between the first and second line of this function we might have an exception and thingy will be destroyed. We want to be ready for “unwinding” the stack.

To atomically replace a value in a way Rust accepts, we need to use std::mem::replace. This moves the second argument into the first, and returns the old value of the first argument for us to use where we need it. This allows us to “steal” thingy without introducing exception un-safety.

Option documentation.

Represents an optional value: either Some value, or None.

Match documentation.

Basically a switch-case with all the fancy syntax you could want.

match on_a_thing {
  an_arm => do_something(),
  another_arm => {
    do_something_else()
  }
}

Useful for the afore-menioned Option type:

match thing {
  Some(thing) => do_something_with(thing),
  None => do_something_else(),
}
  • ! indicates a macro
  • The unimplemented macro:

Indicates unimplemented code by panicking with a message of “not implemented”.

And… Why would I want that?

This allows your code to type-check, which is useful if you are prototyping or implementing a trait that requires multiple methods which you don’t plan to use all of.

The difference between unimplemented! and todo! is that while todo! conveys an intent of implementing the functionality later and the message is “not yet implemented”, unimplemented! makes no such claims. Its message is “not implemented”. So it’s semantic only.

(Also, by reading the code, it looks like todo! is newer.)

Keep your tests in the same file and mark the module with #[cfg(test)] (to indicate we only want to compile this module when we’re running tests).

#[cfg(test)]
mod tests {
  use super::Thingy;

  #[test]
  fn it_works() {
    assert_eq!(2 + 2, Thingy);
  }
}

Then run with cargo test.

Traits are interfaces. :) For example, the Drop trait means “when you go out of scope, you get a chance to cleamup”, and it looks like this:

pub trait Drop {
  fn drop(&mut self);
}
